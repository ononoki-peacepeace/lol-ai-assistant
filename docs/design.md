LOL AI Assistant 设计文档

1. 项目背景

这个项目最初来源于一个想法：能不能为国服《英雄联盟》玩家做一个基于屏幕识别的 AI 辅助分析工具。

目前市面上已有一些战绩查询、符文推荐、出装推荐类工具，但它们大多偏向静态数据分析，或者依赖已有接口数据。我的目标不是做一个自动操作工具，也不是做脚本或外挂，而是尝试做一个更偏“游戏理解”和“意识提醒”的 LOL AI Assistant。

这个项目当前处于早期阶段。第一阶段选择从 BP 识别开始，因为 BP 阶段的信息相对固定，画面结构稳定，适合作为整个项目的第一个视觉识别 MVP。

2. 项目目标

项目的长期目标是构建一个面向国服玩家的 LOL AI Assistant，它可以根据屏幕上的可见信息，辅助玩家理解当前对局状态，并给出意识层面的提醒或建议。

长期希望支持的能力包括：

- BP 阶段识别双方 Ban/Pick 英雄；
- 根据阵容提供简单的阵容理解、强势期和风险提示；
- 记技能，帮助对线；
- 游戏中观察小地图，记录敌方英雄出现和消失，提醒可能来抓了；(中野抓上队友吸血的局我直接)
- 推测敌方打野、中单、辅助的可能动向；
- 根据资源刷新时间提醒小龙、大龙、峡谷先锋等关键节点；
- 根据 Tab 面板中的装备、等级、补刀等信息分析双方战斗力；
- 在合适时机提醒做眼、撤退、支援、控资源；
- 对整局游戏进行赛后复盘，总结关键失误和可改进点。
- 局内通过语音播报输出(二次元音色可行？)

当前阶段并不追求一次性完成全部功能，而是先完成一个可以运行、可以验证思路的最小版本。

3. 项目边界与原则

本项目坚持以下边界：

- 不读取游戏内存；
- 不修改游戏进程；
- 不注入客户端；
- 不自动点击或自动操作；
- 不做脚本、不做外挂；
- 只基于屏幕截图、公开资源和用户手动校准信息进行分析。

项目的定位不是“替玩家操作”，而是“观察屏幕信息并给出辅助理解”。

这也是为什么当前技术路线优先选择屏幕截图、图像识别和规则系统，而不是直接读取游戏内部数据。

---


4. 为什么不直接依赖 Riot API

项目最初考虑过通过 Riot API 获取玩家最近对局、常用英雄等数据。但后来发现，项目主要面向国服玩家，而国服玩家通常使用 WeGame / QQ 体系，与 Riot Developer Portal 的账号体系并不完全一致。

因此当前不把 Riot API 作为核心依赖。

未来如果需要支持外服，可以保留 Riot API 相关模块作为扩展能力。但对于国服方向，项目核心仍然优先基于屏幕识别和公开资源。

---

5. 为什么不一开始就使用大模型实时决策

这个项目长期会和 AI Agent、RAG、本地知识库等方向结合，但当前不打算一开始就让大模型实时判断整局游戏。

原因是实时游戏场景对稳定性和延迟要求很高。如果每秒都把游戏画面交给大模型分析，会遇到几个问题：

1. 延迟较高；
2. 成本较高；
3. 输出不稳定；
4. 容易产生幻觉；
5. 很难保证提醒时机准确。

因此项目更适合采用“快慢结合”的架构：

快脑：规则系统 / 状态机
  - 负责低延迟、稳定、明确的提醒
  - 例如敌方中单消失、资源即将刷新、下路压线风险

慢脑：RAG / Agent / 大模型
  - 负责解释、总结、复盘、复杂阵容分析
  - 例如阵容优劣、对局思路、赛后复盘

也就是说，实时提醒不应该完全依赖大模型，而应该由规则系统和状态维护模块承担。大模型更适合作为解释层和总结层。

6. 后续规则库设想

项目后续计划设计一个本地意识规则库，用来存储高分段玩家的经验和通用游戏理解。

例如：

- 中单消失超过一定时间时，下路压线存在风险；
- 小龙刷新前需要提前布置视野；
- 对方打野长时间未出现时，需要根据野区路线推测位置；
- 己方关键大招未好时，不适合主动接资源团；
- 敌方强开阵容在视野缺失时威胁更高。

规则库可以用 YAML 或 Markdown 存储，例如：

id: missing_mid_bot_danger
name: 中单消失时下路压线风险
trigger:
  enemy_mid_missing_seconds: ">=8"
  enemy_jungle_visible: false
  ally_bot_lane_position: "deep"
advice:
  short: "中单消失较久，下路压线有被抓风险，建议后撤或补河道眼。"
  reason: "敌方中单和打野都未暴露时，下路深位容易被四包二。"
priority: high
confidence: medium

未来希望这个规则库可以持续扩充，甚至通过 GitHub 社区贡献的方式收集更多玩家经验。

7. 未来整体架构设想

长期来看，项目可以扩展为以下结构：

感知层
  - BP 识别
  - 小地图识别
  - Tab 面板识别
  - 资源状态识别
  - 英雄出现 / 消失记录

状态层
  - 敌方英雄最后出现时间
  - 资源刷新时间
  - 双方装备和等级状态
  - 线权、视野、人数差等状态

规则层
  - 本地意识规则库
  - 风险判断
  - 资源团提醒
  - 站位和视野提醒

知识层
  - 本地文档
  - 高分段经验
  - 后续 Web RAG
  - 版本改动、英雄数据、阵容理解

Agent 层
  - 将规则结果转化为自然语言
  - 给出解释
  - 生成赛前建议和赛后复盘

当前项目还处于第一阶段，主要完成感知层中的 BP 识别部分。


---

8. 当前不足
当前项目仍然有很多不足：
英雄头像识别还不够稳定；
空位判断仍需要更多样本；
目前没有 GUI，只能在终端输出；
目前只支持 BP 阶段；
没有接入阵容分析和规则库；
没有实现小地图识别；
项目结构后续可能需要重构；
当前主要是学习和验证性质，还不是完整产品。
这些不足会随着后续开发逐步改进。
